# fiction-revise：章节修改

一次修订 = 定位变更点 → 执行修订 → 探测影响范围 → 传播变更 → 校验一致性。
不能改完一章就完事，系统必须自动回答：这章改了什么，哪些地方也需要改？

## 自动检测推荐

| 检测条件 | 推荐模式 | 说明 |
|---------|---------|------|
| 用户说"润色""太啰嗦""顺一下" | **→ 路由到 fiction-polish** | 只改表达，不改事实 |
| 用户说"把 XXX 改成 YYY""设定不对" | **大改** | 改了事实但主线不变 |
| 用户说"这章重写""推倒重来""重来一遍" | **重写** | 情节走向变了 |
| 修改范围超过一章（"第5-8章都要改"） | **重写** | 波及面广 |
| 修改波及设定集（"世界观改了""主角能力改"） | **重写（广播）** | 需全链路刷新 |

## 大改 模式（核心设计）

`
Step 1: 分析变更
  系统对比新旧版本，提取"变更事实清单"
  ┌── 变更事实清单 ──────────────┐
  │ 1. [人名] 配角 "张三" → "李四"  │
  │ 2. [关系] 主角与张三"敌对"→"盟友"│
  │ 3. [情节] 第7章结尾改写         │
  │ 4. [伏笔] "张三是卧底" → 删除      │
  └────────────────────────────┘

Step 2: 执行修改
  → 修改目标章节正文

Step 3: 影响范围探测（核心步骤）
  系统扫描后续已写章节，检测是否存在引用变更项：
  ┌── 影响范围报告 ──────────────────┐
  │ 🔶 第8章：引用了被改情节 → 需修改 │
  │ 🟡 第9章：提到旧名 1 次 → 换名    │
  │ 🟢 第10章：无关                    │
  │ 🔶 设定集/角色卡/张三.md：需改名     │
  │ 🔶 追踪/伏笔.md：一条伏笔需删除      │
  └────────────────────────────────┘

  → 只读每章摘要做快速过滤，命中再读正文确认
  → 扫描范围控制在最近 10 章以内

Step 4: 用户确认影响范围（展示报告，默认全部选中）

Step 5: 批量执行变更传播
  系统自动：
  → 修改受影响的后续章节（人名替换/情节调整）
  → 更新设定集相关文件
  → 更新追踪系统（伏笔/时间线/角色状态）
  → 更新大纲相关条目
  → 标记被修改的章节为"revised"

Step 6: 全量校验
  → 对受影响章节批量跑轻量 reviewer（只查连续性问题）
  → 一致性检查
  → 伏笔闭环检查
`

## 重写 模式

`
Step 1: 确认范围
  → 单章重写还是多章重写？
  → 章纲是否需要先调整？

Step 2: 可选的细纲修改
  → 如果情节走向变了，先编辑章纲 → 用户确认

Step 3: 执行重写
  → 进入 fiction-write 深度模式完整流程
  → context-agent 重新整理上下文
  → 起草正文

Step 4: 变更传播
  → 同「大改」Step 3-6
  → 范围更广（可能波及拉取后续所有章节）

Step 5: 底层刷新
  → data-agent 重新提取本章事实
  → chapter-commit 覆盖旧 commit
  → 五层投射全部刷新

Step 6: 回填大纲
  → 将最终正文的结构化节点写回章纲
`

## 写回规则（全部静默执行）

修改完成后系统自动写回。写回范围由传播引擎决定，用户不需要逐项确认。

唯一例外：
- 跨卷传播时弹提示："第2卷也可能受影响，是否继续扫描？"
- 文件在上次 git commit 后有用户手动编辑，先问是否保留

## 撤销 / 回滚

- 每次 chapter-commit 或 fiction-revise 执行前自动打 git tag（pre-revise-ch{章节号}）
- 用户不满意可运行：git checkout pre-revise-ch{章节号} {受影响的文件}
- 恢复后重新跑 data-agent 刷新底层
- 这是一个补丁级安全网，不是完整分支管理

## 参考

变更传播引擎依赖每章的摘要文件（.novel/summaries/ch{章节号}.md）做快速过滤。
普通和深度模式下每章写完都会自动生成一句话摘要，确保变更传播始终可快速扫描。
如遇极端情况摘要缺失，退化为逐章读正文判断（慢但可用）。
写回规则与 fiction-write 一致（静默执行，仅跨卷确认）。
轻量 reviewer 同 fiction-review 的 reviewer schema，但只检查连续性问题维度。
章节回滚的安全网由 git tag 提供，不依赖专用脚本。
