## 章节修改

写完之后想改？系统内建两种修改模式，自动检测推荐，不用你自己判断用哪种。

一次修订 = 定位变更点 + 执行修订 + 探测影响范围 + 传播变更 + 校验一致性。
不能改完一章就完事，系统必须自动回答：这章改了什么，哪些地方也需要改？

### 自动检测推荐

| 检测条件 | 推荐模式 | 说明 |
|---------|---------|------|
| 用户说“润色”“太啰嘛”“顺一下” | **→ 路由到 fiction-polish** | 只改表达，不改事实 |
| 用户说“把 XXX 改成 YYY”“设定不对” | **大改** | 改了事实但主线不变 |
| 用户说“这章重写”“推倒重来”“重来一遍” | **重写** | 情节走向变了 |
| 修改范围超过一章 | **重写** | 波及面广 |
| 修改波及设定集 | **重写（广播）** | 需全链路刷新 |

### 大改模式

修改了事实（人名/情节/细节），但主线不变。系统自动检测后续章节影响范围并传播变更。

```
Step 1: 分析变更
  系统对比新旧版本，提取变更事实清单（改了什么名字、什么情节、什么伏笔）

Step 2: 执行修改 → 修改目标章节正文

Step 3: 影响范围探测（核心步骤）
  → 扫描后续已写章节，检测是否存在引用变更项
  → 只读每章摘要做快速过滤，命中再读正文确认
  → 扫描范围控制在最近 10 章以内

Step 4: 用户确认影响范围（展示报告，默认全部选中）

Step 5: 批量执行变更传播
  → 修改受影响的后续章节（人名替换/情节调整）
  → 更新设定集相关文件
  → 更新追踪系统（伏笔/时间线/角色状态）
  → 更新大纲相关条目

Step 6: 全量校验
  → 对受影响章节批量跑轻量 reviewer（只查连续性问题）
  → 一致性检查、伏笔闭环检查
```

### 重写模式

整章推翻重来，情节走向变了。全链路刷新底层和数据层。

```
Step 1: 确认范围
  → 单章重写还是多章重写？章纲是否需要先调整？

Step 2: 可选的细纲修改
  → 如果情节走向变了，先编辑章纲 → 用户确认

Step 3: 执行重写
  → 进入深度模式完整流程
  → context-agent 重新整理上下文 → 起草正文

Step 4: 变更传播 → 同「大改」Step 3-6，范围更广

Step 5: 底层刷新
  → data-agent 重新提取本章事实
  → chapter-commit 覆盖旧 commit

Step 6: 回填大纲 → 将最终正文的结构化节点写回章纲
```

### 撤销 / 回滚

- 每次修改前自动打 git tag
- 不满意可恢复：git checkout pre-revise-ch{N} {受影响的文件}
- 恢复后重新跑 data-agent 刷新底层
